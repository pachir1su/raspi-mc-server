# 라즈베리파이 클러스터 (선택)

Pi 4B 한 대면 3~4명 바닐라식 서바이벌 월드에 충분합니다. 파이 여러 대의
"클러스터"는 **하나의 마인크래프트 월드를 더 빠르게** 만들지 못합니다 —
바닐라/Paper 서버는 월드당 대체로 싱글 스레드라 여러 머신으로 나눌 수 없습니다.
그래서 클러스터는 하나의 월드를 확장하는 수단이 아니라 **더 많은 개별 작업**을
돌리는 수단으로 생각하세요.

## 클러스터가 좋은 경우

- **파이마다 별도 서버**: 예를 들어 Pi #1에 서바이벌, Pi #2에 크리에이티브를 두고
  **프록시**로 묶어 플레이어가 하나의 주소에서 월드를 고르게.
- **부가 서비스 분산**: 디스코드 봇, 백업/rclone, dynmap 같은 웹 맵, 모니터링을
  두 번째 파이에서 돌려 게임 파이는 틱만 처리하게.
- **블루/그린·테스트**: 버전 업그레이드를 월드 이전 전에 시험할 여분 파이 유지.

## 프록시 기반 멀티 서버 (Velocity)

[Velocity](https://papermc.io/software/velocity)는 PaperMC 팀의 현대적 프록시입니다.
플레이어는 프록시에 접속하고, 프록시가 백엔드 Paper 서버(각각 파이 한 대 가능)로
라우팅합니다.

```
                 ┌──────────────┐
플레이어 ───────▶ │  Velocity     │  (Pi #0, 경량)
                 │  프록시 :25565 │
                 └──────┬───────┘
          ┌─────────────┼──────────────┐
          ▼             ▼              ▼
      서바이벌        크리에이티브     미니게임
      (Pi #1)         (Pi #2)         (Pi #3)
```

핵심:

- 백엔드는 **프록시 뒤 offline-mode**로 두고 공유 시크릿으로 Velocity의
  **modern forwarding**을 켜서, 프록시만 접근하게(백엔드는 절대 공개 노출 금지).
- 각 백엔드는 자체 화이트리스트/ops 유지; 각각에서 나만 op.
- 친구 한 월드엔 과합니다 — 여러 월드를 한 주소로 묶고 싶을 때만 쓰세요.

## 가벼운 오케스트레이션

파이 몇 대라면 각 기기의 순수 **systemd**(이 저장소가 설정하는 방식)가
Kubernetes/Docker Swarm보다 간단합니다. 이미 다른 프로젝트로 k3s를 돌린다면
컨테이너화할 수 있지만, 작은 마인크래프트 세팅에선 SD/CPU 오버헤드가 대개
값을 못 합니다.

## 추천

- **친구 3~4명, 한 월드** → Pi 4B 한 대. 클러스터 생략.
- **여러 월드 / 무거운 부가 서비스** → Velocity를 쓰는 파이 2~3대에 부가 서비스를
  분산하고, 전부 systemd로 관리.
